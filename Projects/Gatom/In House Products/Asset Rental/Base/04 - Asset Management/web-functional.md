# Asset Management — Web: Functional Document

> **Product**: Asset Rental Platform
> **Domain**: Asset Management
> **Module**: Customer-Facing Website — Catalog & Asset Detail
> **Document Type**: Functional
> **Audience**: UX designers, frontend developers, QA

---

## 1. Purpose & Scope

This document defines the public asset catalog and detail pages — the primary discovery channel for prospective tenants on the web. These pages are publicly accessible without login and must be optimised for SEO.

---

## 2. Page Requirements

### 2.1 Asset Catalog (`/rentals`)

| # | Requirement |
|---|---|
| WR-001 | All available assets must be browsable without login |
| WR-002 | Results must be filterable by asset type (All / Flats / Vehicles), location, and price |
| WR-003 | Filters must apply without full page reload (AJAX) |
| WR-004 | The first page must be server-rendered for SEO (search engine indexing) |
| WR-005 | Each result must show: cover photo, name, location, key spec (bedrooms or seats), price per month, and an availability badge |
| WR-006 | Results must be paginated |

### 2.2 Asset Detail (`/rentals/{asset}`)

| # | Requirement |
|---|---|
| WR-010 | Photos must be displayable in a carousel with thumbnail strip |
| WR-011 | Key specs must be shown in a visual grid |
| WR-012 | A read-only availability calendar must show which dates are unavailable |
| WR-013 | Unavailable dates must not reveal the reason (maintenance, other rentals, and inspections are all opaque) |
| WR-014 | If a 3D preview is configured, a button must open it in an iframe |
| WR-015 | The deposit amount must be clearly visible before booking |
| WR-016 | A "Request to Book" CTA must be shown for logged-in users with KYC status `Verified` |
| WR-016a | For logged-in users whose KYC status is NOT `Verified`, the "Request to Book" CTA must be replaced with a "Complete Verification" prompt, linking to `/my-kyc` |
| WR-017 | A "Login to Book" CTA must be shown for guests, linking to login with a return redirect |
| WR-018 | The page must have unique `<title>`, `<meta description>`, and `og:image` for SEO |

---

## 3. User Stories

| ID | As a... | I want to... | So that... |
|---|---|---|---|
| WS-001 | Anonymous visitor | Browse available assets without logging in | I can evaluate options before committing |
| WS-002 | Anonymous visitor | See which dates are blocked on an asset | I can plan my rental period |
| WS-010 | Google | Index the asset catalog pages | The client's listings appear in search results |

---

## 4. Business Rules

1. The catalog must serve a server-rendered HTML page on first load (SEO requirement).
2. After the first page, filter interactions use AJAX — no full page reload.
3. Blocked dates on the availability calendar must **never reveal the reason**.

---

## 5. SEO Requirements

| Element | Requirement |
|---|---|
| `<title>` | Unique per page — asset name for detail, "Browse Rentals" for catalog |
| `<meta description>` | ≤ 160 chars, unique per asset detail page |
| `og:image` | First asset photo used as Open Graph image |
| `og:type` | `product` for asset detail pages |
| Server-side render | Catalog first page and all detail pages must be fully rendered server-side |
| Lazy loading | All images use `loading="lazy"` |
| Canonical URL | `<link rel="canonical">` on all asset detail pages |
| Sitemap | Auto-generated by Frappe for all public `www/` pages |
| robots.txt | Catalog indexed; portal pages disallowed |

---

## 6. Security Requirements

| Requirement | Description |
|---|---|
| **Availability opacity** | `get_asset_availability` returns only date ranges, never the reason |
