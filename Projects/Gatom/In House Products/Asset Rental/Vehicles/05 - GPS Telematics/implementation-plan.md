## 1. Overview

GPS Telematics provides **real-time vehicle tracking** for GPS-equipped fleet vehicles. The domain handles: inbound GPS event ingestion via webhook, high-performance raw event storage (bypassing Frappe ORM for write speed), live position streaming via Frappe Socketio, speed/geofence alerting, and data lifecycle management (90-day raw event retention with daily aggregation).

GPS telematics is fundamentally different from every other domain in the system because of its **data volume**. A single GPS device sends a position event every 10–30 seconds. A fleet of 100 GPS-equipped vehicles generates 4,000–10,000 events per hour. This volume makes the standard Frappe ORM (which creates a full DocType record with audit logs, change tracking, and permission checks for each insert) prohibitively expensive.

This domain handles: (1) **inbound GPS event ingestion** via an HMAC-secured webhook that GPS hardware providers (Teltonika, Queclink, etc.) call when a device reports its position, (2) **high-performance raw event storage** in a custom SQL table (bypassing Frappe's ORM entirely), (3) **live position streaming** via Frappe Socketio for real-time Fleet Manager dashboards, (4) **speed/geofence alerting** for safety monitoring, and (5) **data lifecycle management** — raw events are aggregated daily and purged after 90 days to prevent unbounded table growth.

This is a **Fleet Manager-only feature** — customers never see GPS data directly (knowing they're being tracked creates liability issues). The only customer-facing impact is that GPS data feeds the **estimated mileage** calculation in V03 — allowing customers with GPS-equipped vehicles to see how many km they've driven mid-rental.

## 2. Frappe — Raw SQL Table

### 2.1 `tabVehicle GPS Event` Table Creation

> **Requires**: V01-2.1 (app scaffold), database migration mechanism

The **GPS event table** is created via a raw SQL migration — NOT through Frappe's DocType framework. This is an intentional architectural decision: Frappe's DocType system adds significant overhead per insert (permission checks, workflow triggers, change logs, naming series), which is unacceptable at telemetry scale. A direct SQL `INSERT` takes <1ms; a DocType `.insert()` takes 50–100ms. At 10,000 events/hour, that's the difference between a 10-second batch and a 15-minute batch.

The table stores the minimal data needed for each GPS ping: device ID (from the hardware), asset link (resolved from the device ID), latitude/longitude (at DECIMAL(10,7) precision — 11cm accuracy, more than sufficient for vehicle tracking), speed in km/h, heading in degrees, and UTC timestamp.

Two indexes are critical: `(asset, timestamp)` enables efficient time-range queries ("show me all positions for vehicle X from March 5–10"), and `(device_id, timestamp)` enables device-based lookups ("which vehicle does device D12345 belong to?"). Without these indexes, queries against a table with millions of rows would be impractically slow.

The migration script runs during `bench migrate` or can be executed manually via `bench execute`. It uses `CREATE TABLE IF NOT EXISTS` to be idempotent — running it twice doesn't error.

```sql
CREATE TABLE IF NOT EXISTS `tabVehicle GPS Event` (
    `name`      VARCHAR(20)  PRIMARY KEY,
    `device_id` VARCHAR(50)  NOT NULL,
    `asset`     VARCHAR(140),
    `lat`        DECIMAL(10,7),
    `lng`        DECIMAL(10,7),
    `speed_kmh`  SMALLINT,
    `heading`    SMALLINT,
    `timestamp`  DATETIME     NOT NULL,
    INDEX idx_asset_ts (`asset`, `timestamp`),
    INDEX idx_device_ts (`device_id`, `timestamp`)
) ENGINE=InnoDB;
```

**Acceptance Criteria**:
- [ ] Table created successfully via migration script
- [ ] `name` is a short hash primary key (generated by `frappe.generate_hash(8)`)
- [ ] `lat` and `lng` use `DECIMAL(10,7)` for 11cm precision
- [ ] `speed_kmh` is `SMALLINT` (max 65535 — sufficient for vehicle speed)
- [ ] Index on `(asset, timestamp)` exists for time-range queries
- [ ] Index on `(device_id, timestamp)` exists for device-based lookups
- [ ] Table does NOT appear in Frappe's DocType registry (it's raw SQL)

### 2.2 `gps_daily_summary` Aggregation Table

> **Requires**: 2.1

Raw GPS events are high-volume, short-lived data — after a few days, individual pings ("vehicle was at 25.2764, 55.2962 at 14:32:07") have limited value. What matters for long-term analysis is the **daily aggregate**: how far did the vehicle drive today, what was the max speed, and where did it start/end the day. This aggregation serves three purposes:

1. **Storage management**: 90 days of raw events for 100 vehicles = ~26 million rows. Aggregating to daily summaries reduces this to ~9,000 rows for the same period.
2. **Mileage estimation**: The V03 mileage domain uses daily summary `total_km` values (not raw events) to estimate current mileage for GPS-equipped vehicles.
3. **Fleet analytics**: The Fleet Manager can see utilization patterns ("Vehicle X drove 8,000 km last month") without querying millions of raw events.

The summary includes: total km driven (computed using the Haversine formula between sequential GPS events), max speed, average speed, event count, and the start/end coordinates for the day. The unique constraint on `(vehicle, summary_date)` ensures exactly one summary row per vehicle per day.

```sql
CREATE TABLE IF NOT EXISTS `tabGPS Daily Summary` (
    `name`         VARCHAR(20) PRIMARY KEY,
    `vehicle`      VARCHAR(140) NOT NULL,
    `summary_date` DATE NOT NULL,
    `total_km`     FLOAT DEFAULT 0,
    `max_speed`    SMALLINT DEFAULT 0,
    `avg_speed`    FLOAT DEFAULT 0,
    `start_lat`    DECIMAL(10,7),
    `start_lng`    DECIMAL(10,7),
    `end_lat`      DECIMAL(10,7),
    `end_lng`      DECIMAL(10,7),
    `event_count`  INT DEFAULT 0,
    UNIQUE KEY uk_vehicle_date (`vehicle`, `summary_date`)
) ENGINE=InnoDB;
```

**Acceptance Criteria**:
- [ ] Table created via migration
- [ ] Unique constraint on `(vehicle, summary_date)` prevents duplicate summaries
- [ ] `total_km` computed from GPS event sequences using Haversine formula
- [ ] `max_speed` and `avg_speed` computed from event `speed_kmh` values
- [ ] Start/end coordinates taken from first and last events of the day

## 3. Frappe — GPS Webhook Endpoint

### 3.1 `receive_gps_event` Webhook (`api/telematics.py`)

> **Requires**: 2.1, V01-2.3 (`custom_gps_device_id` for device-to-asset mapping), Base D01-3.1 (`gps_webhook_secret` in config)

This is the **public-facing HTTP endpoint** that receives GPS position data from hardware providers. GPS tracking devices don't call Frappe APIs — they send HTTP POST requests to a configured URL with a JSON payload containing the device ID, position, speed, and timestamp. This webhook is that URL.

The request lifecycle is: (1) **parse the JSON payload**, (2) **verify the HMAC signature** (to prevent spoofed data — see 3.2), (3) **resolve the device ID to a Rental Asset** by looking up `custom_gps_device_id` on all Vehicle assets, (4) **insert directly into the raw SQL table** using `frappe.db.sql()` (not the ORM), and (5) **publish the event to Frappe Socketio** on channel `gps:{asset_name}` so the live map updates in real-time.

The endpoint is `allow_guest=True` because GPS hardware devices don't authenticate with Frappe sessions — they use HMAC signatures instead. The `frappe.db.commit()` after insertion is explicit (not relying on Frappe's auto-commit at request end) to ensure the data is persisted immediately.

Unknown device IDs (devices not yet linked to a vehicle in the system) are **not rejected** — the event is stored with `asset = NULL`. This is important because a Fleet Manager might install a GPS device on a new vehicle before adding it to the system. The events are still captured and can be retroactively linked when the vehicle is registered.

```python
@frappe.whitelist(allow_guest=True, methods=["POST"])
def receive_gps_event():
    payload = frappe.request.get_json()
    _verify_hmac(payload, frappe.get_single("Rental Configuration").gps_webhook_secret)
    frappe.db.sql("""
        INSERT INTO `tabVehicle GPS Event`
        (name, device_id, asset, lat, lng, speed_kmh, heading, timestamp)
        VALUES (%s, %s, %s, %s, %s, %s, %s, %s)
    """, (
        frappe.generate_hash(8),
        payload["device_id"],
        _resolve_asset(payload["device_id"]),
        payload["lat"], payload["lng"],
        payload.get("speed", 0),
        payload.get("heading", 0),
        payload["ts"],
    ))
    frappe.db.commit()
```

**Acceptance Criteria**:
- [ ] Endpoint accessible at `/api/method/rental_vehicles.api.telematics.receive_gps_event`
- [ ] Publicly accessible (no Frappe auth — webhooks come from GPS hardware)
- [ ] **HMAC signature validated** — invalid signature → HTTP 401 + log to `GPS Webhook Failure Log`
- [ ] Valid event → row inserted into `tabVehicle GPS Event`
- [ ] `device_id` resolved to Rental Asset via `custom_gps_device_id` lookup
- [ ] Unknown `device_id` → event stored with `asset = NULL` (not rejected)
- [ ] Missing `gps_webhook_secret` in config → `frappe.throw` with setup instructions
- [ ] Event published to Socketio channel `gps:{asset_name}` for live map

### 3.2 HMAC Verification (`_verify_hmac`)

> **Requires**: 3.1, Base D01-3.1 (config)

Without HMAC verification, anyone who knows the webhook URL could send fake GPS data — making it appear that a vehicle is at a location it's not, or injecting false speed readings that trigger alerts. The HMAC verification prevents this by ensuring only the legitimate GPS provider can send valid events.

The verification works by: (1) the GPS provider computes an HMAC-SHA256 of the request body using a shared secret (the `gps_webhook_secret` stored in Rental Configuration), (2) the provider includes this signature in the `X-Signature` HTTP header, (3) the webhook recomputes the HMAC from the received body using the same secret, and (4) compares the two values using a **timing-safe comparison** (to prevent timing attacks that could leak the secret byte-by-byte).

If the signatures don't match, the webhook returns HTTP 401 and logs the failure in the `GPS Webhook Failure Log` DocType (3.3). If more than 5 failures occur within an hour, an alert notification is sent to the Fleet Manager — this could indicate an attack or a misconfigured GPS device.

**Acceptance Criteria**:
- [ ] HMAC computed from request body using SHA-256 with `gps_webhook_secret`
- [ ] Signature from `X-Signature` header compared against computed HMAC
- [ ] Mismatch → HTTP 401 response
- [ ] Failure logged in `GPS Webhook Failure Log` DocType
- [ ] More than 5 failures/hour → alert notification sent to Fleet Manager
- [ ] Timing-safe comparison used (prevents timing attacks)

### 3.3 `GPS Webhook Failure Log` DocType

> **Requires**: V01-2.1 (app scaffold)

A simple, append-only logging DocType for tracking GPS webhook authentication failures. This serves two purposes: (1) **security monitoring** — a sudden spike in failures might indicate someone is trying to inject fake GPS data, and (2) **debugging** — when a legitimate GPS device isn't sending data properly, the failure log shows exactly what went wrong (wrong signature, missing header, etc.).

The log captures: the device ID from the payload (to identify which device is failing), the source IP address (to trace the request origin), the failure reason ("HMAC mismatch", "Missing signature header", etc.), and the first 500 characters of the payload (for debugging, without storing the entire potentially-large payload).

The log is **immutable** — read-only for all roles, set programmatically. This ensures the audit trail cannot be tampered with.

| Field | Type | Notes |
|---|---|---|
| `device_id` | Data | From the payload |
| `source_ip` | Data | Request IP address |
| `failure_reason` | Data | "HMAC mismatch", "Missing signature", etc. |
| `payload_excerpt` | Text | First 500 chars of the payload |
| `created_at` | Datetime | Auto |

**Acceptance Criteria**:
- [ ] Every failed HMAC validation creates a log entry
- [ ] Log is read-only for all roles (immutable audit trail)
- [ ] Only Fleet Manager and System Manager can view
- [ ] `source_ip` captured from `frappe.request.remote_addr`

## 4. Frappe — Speed & Geofence Alerting

### 4.1 Speed Threshold Alert

> **Requires**: 3.1, Base D01-3.1 (config: `gps_speed_threshold_kmh`), Base D07 (notification system)

Excessive speed is a safety concern and a liability risk — if a rented vehicle is involved in a speed-related accident, the rental company needs to demonstrate that it monitored driver behavior. This alert checks each incoming GPS event's speed against a configurable threshold (e.g., 140 km/h) and immediately notifies the Fleet Manager if the threshold is exceeded.

The check runs **inline during event ingestion** (not as a batch job) to ensure real-time alerting. When a GPS event arrives with `speed_kmh > gps_speed_threshold_kmh`, a notification is sent immediately. The alert includes the vehicle name ("Toyota Camry ABC-1234"), the recorded speed ("157 km/h"), and the GPS coordinates (so the Fleet Manager can see where the speeding occurred).

**Rate limiting** is critical here: a vehicle doing 160 km/h on a highway will send multiple events above the threshold (one every 10–30 seconds). Without rate limiting, the Fleet Manager would receive dozens of identical alerts for a single speeding incident. The system enforces a maximum of **1 alert per vehicle per 15 minutes** — the first event above threshold triggers the alert, then subsequent events from the same vehicle are silently recorded for the next 15 minutes.

If no threshold is configured in Rental Configuration, the speed alert feature is effectively disabled (no alerts fire).

**Acceptance Criteria**:
- [ ] Speed above `gps_speed_threshold_kmh` → notification sent to Fleet Manager
- [ ] Speed below threshold → no alert
- [ ] Alert includes: vehicle name, speed, lat/lng, timestamp
- [ ] Alert rate-limited: max 1 alert per vehicle per 15 minutes (prevents alert storms from sustained speeding)
- [ ] No threshold configured → no speed alerts (feature disabled)

### 4.2 Geofence Violation Logging

> **Requires**: 3.1

A **v1 stub** for geofence violation detection. A geofence is a virtual geographic boundary — for example, the Fleet Manager might define a geofence around a city ("vehicles must stay within Dubai") or around a specific area ("do not enter the airport restricted zone"). When a GPS-equipped vehicle crosses the boundary, the system logs the violation and notifies the Fleet Manager.

In v1, this is **logging only** — no automatic action is taken (the system doesn't remotely disable the vehicle or block the agreement). The Fleet Manager defines geofence boundaries as lat/lng polygons via Desk, and violations are recorded for review. This is a foundation for future features like geo-restricted rental zones or automatic toll calculations.

**Acceptance Criteria**:
- [ ] Geofence boundaries definable per vehicle or per category (lat/lng polygon)
- [ ] Event outside boundary → geofence violation logged
- [ ] Notification sent to Fleet Manager
- [ ] v1: logging only (no automatic action taken)

## 5. Frappe — Data Lifecycle Management

### 5.1 Daily Aggregation Scheduler

> **Requires**: 2.1, 2.2

Raw GPS events are only useful in real-time (for live tracking) and for short-term analysis. After a few days, individual pings have no business value — nobody cares that "Vehicle X was at latitude 25.2764 at 14:32:07 on January 5." What matters is the daily aggregate: "Vehicle X drove 287 km on January 5 with a max speed of 128 km/h."

This scheduler runs **daily at 2 AM** (off-peak) and processes the **previous day's** events (not the current day, because events may still be arriving). For each vehicle that has GPS events from yesterday, it: (1) sorts events by timestamp, (2) computes total km using the **Haversine formula** between each pair of sequential events (the great-circle distance between two lat/lng points), (3) finds the max and average speed from the event `speed_kmh` values, (4) counts total events, and (5) takes the first event's coordinates as `start_lat/lng` and the last event's as `end_lat/lng`.

The summary uses `INSERT ... ON DUPLICATE KEY UPDATE` (or equivalent) so that re-running the aggregation for the same vehicle+date doesn't create duplicates — it updates the existing row.

**Acceptance Criteria**:
- [ ] Runs daily at 2 AM
- [ ] Aggregates events from the previous day only (not current day — events may still be arriving)
- [ ] `total_km` computed using Haversine formula between sequential events
- [ ] `max_speed` and `avg_speed` computed from event `speed_kmh`
- [ ] `start_lat/lng` from first event; `end_lat/lng` from last event
- [ ] Duplicate summary for same vehicle+date → update (not duplicate insert)
- [ ] Vehicles with no events yesterday → no summary row created

### 5.2 90-Day Raw Event Purge Scheduler

> **Requires**: 2.1, 5.1 (aggregation must run first)

Without a purge mechanism, the raw GPS event table grows without bound. 100 GPS devices × 1 event every 15 seconds × 86,400 seconds/day = ~576,000 rows/day. Over a year, that's 210 million rows — making the table unusable for queries and consuming significant storage.

The purge scheduler runs **daily at 3 AM** (one hour after aggregation, to ensure the daily summary has been computed first). It deletes all raw GPS events with `timestamp < NOW() - 90 days`. The 90-day retention window is chosen as a balance between operational needs (investigating recent incidents, debugging GPS device issues) and storage efficiency.

**Critically**: the daily summary data (computed by 5.1) is retained indefinitely. So even after raw events are purged, historical questions like "how far did Vehicle X drive last year?" can still be answered from the summary table. The purge only affects the high-volume raw event table.

**Acceptance Criteria**:
- [ ] Events older than 90 days are deleted from `tabVehicle GPS Event`
- [ ] Purge runs daily (after aggregation at 2 AM — e.g., at 3 AM)
- [ ] Count of deleted events logged for auditing
- [ ] Purge runs without error when no old events exist
- [ ] Daily summaries are NOT affected by the purge

## 6. Frappe — Live Map API

### 6.1 `get_vehicle_location` API

> **Requires**: 2.1, V01-2.3 (`custom_has_gps`)

This API endpoint powers the **Vehicle Map** screen in the Flutter app. When the Fleet Manager taps on a specific vehicle to see its current position, this endpoint returns the **most recent GPS event** for that vehicle — the latest latitude, longitude, speed, heading, and timestamp from the raw event table.

The endpoint is **role-restricted to Fleet Manager only**. Customers never see GPS data — this is both a privacy boundary (customers shouldn't know they're being tracked in real-time) and a liability protection (if the GPS data is inaccurate and a customer relies on it, the company could face issues). Rental Agents are also excluded — live vehicle tracking is a fleet management function, not a counter operation.

For vehicles without GPS (`custom_has_gps = 0`), the endpoint returns an empty response with a descriptive message. For GPS vehicles with no events yet (device just installed), a different message is returned.

**Acceptance Criteria**:
- [ ] Returns: `lat`, `lng`, `speed_kmh`, `heading`, `timestamp` from the latest GPS event
- [ ] Fleet Manager role → data returned
- [ ] Customer role → HTTP 403
- [ ] Rental Agent role → HTTP 403
- [ ] Vehicle without GPS (`custom_has_gps = 0`) → empty response with message "No GPS device linked"
- [ ] Vehicle with GPS but no events → empty response with message "No position data available"

### 6.2 `get_all_vehicle_locations` API

> **Requires**: 6.1

The Fleet Map needs to show **all** GPS-equipped vehicles simultaneously — not just one at a time. This endpoint returns a list of the last known position for every GPS-equipped vehicle in the fleet. The response includes: asset name (for map marker labels), latitude, longitude, speed, and timestamp.

This is what powers the "bird's eye view" of the fleet — the Fleet Manager opens the map and sees 50 markers scattered across the city, each showing where a vehicle currently is. Clicking a marker shows the vehicle's name and current speed.

The endpoint only returns vehicles with `custom_has_gps = 1` AND at least one GPS event in the system. GPS vehicles that have been registered but haven't yet reported their first position are excluded (no point showing a marker with no location data). Fleet Manager role only.

**Acceptance Criteria**:
- [ ] Returns list of `{asset_name, lat, lng, speed_kmh, timestamp}` for all GPS vehicles
- [ ] Only includes vehicles with `custom_has_gps = 1` and at least one GPS event
- [ ] Fleet Manager role only
- [ ] Sorted by asset name

## 7. Flutter — Live Fleet Map

### 7.1 Vehicle Map Screen (`widgets/vehicle_map_screen.dart`)

> **Requires**: 6.2 (all vehicle locations API), Base D01-8.2 (google_maps_flutter dependency)

The **Fleet Map** is a full-screen Google Map showing the real-time positions of all GPS-equipped vehicles in the fleet. Each vehicle appears as a **marker** on the map with the vehicle's name. ### 7.2 Live Position Auto-Refresh

> **Requires**: 7.1

Static map markers don't tell the Fleet Manager where a vehicle is *now* — only where it was when the data was last fetched. For real-time fleet monitoring, the markers need to **move** as vehicles move. This subtask implements two mechanisms for keeping the map current:

**Primary: Frappe Socketio** — the Flutter app subscribes to Socketio channels for each vehicle (one channel per vehicle: `gps:{asset_name}`). When the GPS webhook (3.1) receives a new event, it publishes the position to the channel, and the Flutter app receives it instantly via WebSocket. The marker animates smoothly to the new position using a `Tween` animation (so it appears to slide rather than jump).

**Fallback: 10-second polling** — in environments where WebSocket connections are unreliable (e.g., corporate firewalls, unstable mobile networks), the app falls back to polling the `get_all_vehicle_locations` API every 10 seconds. This is detected automatically: if the Socketio connection fails or no data arrives for 30 seconds, the app switches to polling mode.

**Stale data indicator**: If a vehicle hasn't reported a position in more than 5 minutes (the device might be offline, out of cellular range, or malfunctioning), the marker is grayed out with a tooltip: "Last seen: 12 min ago." This prevents the Fleet Manager from assuming a stale marker represents the vehicle's current position.

**Acceptance Criteria**:
- [ ] Socketio connection: markers update in real-time as events arrive
- [ ] Socketio fallback: 10-second polling via API call
- [ ] Markers animate to new positions (smooth transition)
- [ ] Stale data (no update in 5 minutes) → marker grayed out with "Last seen: X min ago"

### 8.2 Caching

GPS data has unique caching requirements due to real-time nature and high volume:

| Data | Cache Key Pattern | TTL | Invalidation Trigger |
|---|---|---|---|
| Last known position (per vehicle) | `gps_last_pos:{asset_name}` | 60 sec | New GPS event received for that vehicle |
| All vehicle positions (fleet map) | `gps_fleet_positions:all` | 10 sec | Any new GPS event |
| Device-to-asset mapping | `gps_device_map:{device_id}` | 5 min | Rental Asset save (GPS device ID change) |
| Speed threshold config | `gps_speed_threshold` | 30 min | Rental Configuration save |
| Daily summary (per vehicle) | `gps_daily_summary:{vehicle}:{date}` | 24 hr | Daily aggregation scheduler run |

**Implementation**: Use Frappe's `frappe.cache()` (Redis-backed). The device-to-asset mapping cache is critical for webhook performance — without it, every incoming GPS event would require a DB lookup to resolve `device_id` → `asset_name`.

**Acceptance Criteria**:
- [ ] Device-to-asset mapping cached with 5-min TTL to avoid per-event DB lookups
- [ ] Fleet map positions cached with very short TTL (10 sec) for near-real-time updates
- [ ] Speed threshold cached to avoid per-event config lookups
- [ ] Daily summary cached after aggregation for downstream V03 mileage estimate queries

### 8.3 Rate Limiting

The GPS webhook is a public endpoint — rate limiting is critical to prevent abuse:

| Endpoint | Limit | Scope | Response on Limit |
|---|---|---|---|
| `receive_gps_event` webhook | 1000 req/min | Per source IP | HTTP 429 (silent — GPS devices will retry) |
| `get_vehicle_location` API | 60 req/min | Per user (Fleet Manager) | HTTP 429 with `Retry-After` header |
| `get_all_vehicle_locations` API | 30 req/min | Per user (Fleet Manager) | HTTP 429 with `Retry-After` header |
| Aggregation scheduler | N/A (server-side) | Daily at 2 AM | Not applicable |
| Purge scheduler | N/A (server-side) | Daily at 3 AM | Not applicable |

**Implementation**: Webhook rate limiting at Nginx layer (`limit_req_zone` by source IP) for performance — Frappe-level rate limiting would add latency to the high-volume webhook. API endpoints use Frappe's `frappe.rate_limiter.rate_limit()` decorator.

**Acceptance Criteria**:
- [ ] Webhook rate-limited at Nginx level (not Frappe) for minimal latency
- [ ] Fleet map APIs rate-limited to prevent excessive polling
- [ ] Rate limit on webhook does not cause GPS data loss (devices retry on 429)
- [ ] Scheduler jobs protected by Frappe's built-in scheduler lock

| Source IP logging | All webhook requests | `frappe.request.remote_addr` captured for security audit trail |

| Customer GPS data prohibition | All customer-facing surfaces | Customers NEVER see GPS data (privacy + liability protection) |
| Payload validation | `receive_gps_event` | Required fields: `device_id`, `lat`, `lng`, `ts`; reject malformed payloads |
| Coordinate bounds validation | `receive_gps_event` | Latitude: -90 to 90; Longitude: -180 to 180; reject out-of-bounds values |
| Speed bounds validation | `receive_gps_event` | Speed: 0 to 500 km/h; reject negative or impossibly high values |
| SQL injection prevention | Raw SQL inserts | Use parameterized queries for all `tabVehicle GPS Event` inserts |
| Failure log immutability | `GPS Webhook Failure Log` | Read-only for all roles; set programmatically only |
| Payload excerpt sanitization | Failure log | Truncate payload to 500 chars; strip any HTML/script tags |
| Socketio channel authorization | `gps:{asset_name}` | Only Fleet Manager sessions can subscribe to GPS channels |
